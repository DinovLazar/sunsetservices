## 0. Read this first

Three surfaces, one shared design language:

1. **Privacy + Terms pages** at `/privacy/`, `/terms/`, `/es/privacy/`, `/es/terms/`. Both share one layout template — utility chrome, narrower prose container, sticky TOC on desktop / accordion on mobile, Termly embed rendering the policy body.
2. **Cookie consent banner** — sitewide, bottom-fixed, first-visit only, Google Consent Mode v2 wired.
3. **Preferences modal** — opens from the banner's "Manage preferences" button. Four toggles mapped to CMv2 signals.

**Visual mockups for every surface live in `docs/design-handovers/Phase-B-02-Legal-Mockups.html`** — open it side-by-side with this document. It shows desktop + mobile, EN + ES, plus a states reference card. The mockups are HTML/CSS so Code can inspect any element to extract padding, font sizes, radii directly.

**No new tokens. No new component variants.** Everything below references a locked token name from Phase 1.03. The cookie banner reuses the existing Primary green / Secondary bordered green / Ghost button trio at `md` (48h). The preferences modal is the existing `card` system at `--radius-lg` with `--shadow-card` inside a `--color-overlay-50` scrim. The page template is the locked `ProseLayout` + a small additive `LegalTocSidebar` component spec'd in §1.

**No backend.** Code drops in the Termly script in B.03; Termly's wizard generates the actual policy text; nothing on this site stores PII for legal-page rendering. Consent state lives in `localStorage` under one key (`sunset_consent_v2`); the four CMv2 signals (`ad_storage`, `analytics_storage`, `ad_user_data`, `ad_personalization`) are persisted and read by Google Consent Mode v2 in B.03.

The 1.06 bar still holds: **professional, no shortcuts**, WCAG 2.2 AA, Lighthouse ≥95 P/A/BP/SEO desktop AND mobile. Two risks unique to this phase:

- **Termly's embed brings its own CSS.** It can drift from our type scale and link colors. The override block in §3.4 below pins it back to locked tokens.
- **Implicit consent is illegal under GDPR-style frameworks.** ESC does not dismiss the banner. Clicking the scrim behind the modal does not dismiss it. The only paths off the banner are Accept all, Reject all, or Save preferences. Spelled out explicitly in §4.7.

### 1.3 Locked from earlier phases — DO NOT redesign

- **Components** (1.03 §6) — Button (Primary green / Secondary bordered / Ghost / Amber — note Amber is NOT used on the banner), Card (`.card`, `.card--cream`), form fields with their full state set, Switch base-ui primitive wrapped in `src/components/ui/switch.tsx`.

- **`ProseLayout`** (1.17 §13) — used unchanged for the policy body container.
- **`MotionConfig reducedMotion="user"`** mounted at the locale-level layout (1.03 §7.7) — every motion spec below assumes this and does not re-declare it.

### 4.4 Banner — motion

| Property | Value |
|---|---|
| Library | `motion/react` only (1.03 §7) |
| Trigger | First visit (`useConsent().hasDecided === false`) AND `document.readyState === 'complete'` |
| Delay | **600ms** after load — keeps clear of the hero entrance choreography |
| Entry — desktop & mobile | Spring slide-up from `y: 100%` to `y: 0`, opacity `0 → 1` |
| Spring config | `type: 'spring', stiffness: 240, damping: 28, mass: 0.9` — produces a settled landing without an overshoot wobble. Total duration ~520ms. |
| Reduced motion | `MotionConfig reducedMotion="user"` already mounted at root (1.03 §7.7). Motion respects user OS preference automatically. With reduced-motion: opacity-only fade-in, 220ms (`--motion-base`), `--easing-standard`. No vertical translation. |
| Exit | After Accept all / Reject all / Save preferences (modal): spring slide-down `y: 100%`, opacity → 0, 320ms. Then component unmounts and never re-renders this session unless localStorage is cleared. |

### 4.5 Banner — accessibility

| Concern | Spec |
|---|---|
| Role | `role="dialog"` on the banner root |
| `aria-labelledby` | Points to the headline `<h3>` id (e.g., `cookie-banner-headline`) |
| `aria-describedby` | Points to the body `<p>` id (e.g., `cookie-banner-body`) |
| Focus trap | Active while banner is shown. Tab cycles between the three buttons + privacy link only. (`focus-trap-react` already in the dependency set from 1.15.) |
| Initial focus | On the **headline element** (`tabIndex={-1}` + `.focus()` on mount). Rationale: lands the user's attention on context before action — they choose where to go next. Alternate considered: focus on the first button. Rejected — pre-selects an action before the user reads. |
| Keyboard | Tab order: Headline → body → Manage preferences (ghost) → Reject all → Accept all → Privacy link → cycle. ESC: **does not dismiss** (no implicit consent). |
| Touch targets | Every button ≥ 48×48 px (visual + hitbox). Privacy link is text-link; surrounding `<p>` carries 44px tap height. |
| Contrast pairs | Body text (#4A4A4A) on cream (#FAF7F1): **8.4 : 1** ✓ (Phase 1.03 contrast row 6). Headline (#1A1A1A) on cream: **17.7 : 1** ✓. Green-500 primary button vs cream surrounding: **3.5 : 1** ✓ non-text. Border-strong top divider on cream: **3.1 : 1** ✓ non-text. |
| Focus ring | `2px solid var(--color-focus-ring)` (#6FA85F), offset 2px. On the primary-green button, ring switches to `--color-text-on-green` (white) so it clears the green field — per locked Decision D1 (1.03 §2.4). |

### 4.6 Banner — state persistence

```ts
// src/lib/consent/types.ts
export type ConsentSignals = {
  ad_storage:          'granted' | 'denied';
  analytics_storage:   'granted' | 'denied';
  ad_user_data:        'granted' | 'denied';
  ad_personalization:  'granted' | 'denied';
};

export type ConsentRecord = {
  v: 2;                              // schema version — bump if categories change
  signals: ConsentSignals;
  decided_at: string;                // ISO timestamp
  locale: 'en' | 'es';               // locale at decision time (for audit)
};
```

| Action | Resulting signals |
|---|---|
| **Accept all** | All four `'granted'` |
| **Reject all** | All four `'denied'` |
| **Save preferences** (modal) | Per the four toggles: necessary is implicit (not a CMv2 signal); analytics → `analytics_storage`; marketing → `ad_storage`; personalization → BOTH `ad_user_data` AND `ad_personalization` |

Storage key: **`sunset_consent_v2`**, written to `localStorage`. Banner reads this on mount via `useConsent()`; if present and parses as `{ v: 2, ... }`, banner does not render. If absent OR schema-version mismatch, banner shows.

Google Consent Mode v2 update fires on every state write: `gtag('consent', 'update', { ... })` — wired in B.03.

### 4.7 No-implicit-consent paths

This is locked. Every path off the banner must be an explicit choice.

| Attempt | Banner behavior |
|---|---|
| Click "Accept all" | All four signals `granted`, record persisted, banner exits, CMv2 update fires. |
| Click "Reject all" | All four signals `denied`, record persisted, banner exits, CMv2 update fires (with denied values). |
| Click "Manage preferences" | Opens modal. Modal must be Save'd or Cancel'd to return to banner; banner does not auto-dismiss. |
| Press ESC | Nothing. Banner stays. |
| Click somewhere on the page (outside banner) | Nothing. Focus stays trapped in banner. |
| Refresh the page | Banner re-shows (`hasDecided` still `false`). |
| Resize the viewport / orientation change | Banner adapts layout. No dismiss. |
| Open preferences modal, press ESC | Modal closes; banner remains, focus returns to "Manage preferences" button. |
| Open preferences modal, click scrim | Nothing. Modal stays. |

### 4.10 The four toggle rows

Each row is a grid: `1fr auto` (description block | switch). 16px gap. 16px vertical padding. Hairline divider between rows: `border-top: 1px solid --color-border`.

| # | Category (EN / ES) | Description (EN — first-pass) | Description (ES — first-pass) | CMv2 signal(s) | Default | Switch state |

Aggregated, never tied to your identity." | "Nos ayuda a entender qué páginas son útiles y dónde el sitio se traba. | 3 | Marketing / Marketing | "Allows ads on other sites to know whether our ads led you here, so we can stop running ones that don't work." | "Permite que los anuncios en otros sitios sepan si nuestros anuncios te trajeron aquí, para dejar de mostrar los que no funcionan." | `ad_storage` | **off** | togglable |

### 4.11 Modal — motion

| Property | Value |
|---|---|
| Open — modal | Scale `0.95 → 1.0` + opacity `0 → 1`, 260ms, `--easing-standard` (cubic-bezier(0.2, 0.0, 0.2, 1)) |
| Open — scrim | Opacity `0 → 1`, 220ms, `--easing-standard`. Fades independently of the modal — backdrop appears slightly before the modal lands. |
| Close — modal | Scale `1.0 → 0.97` + opacity `1 → 0`, 180ms, `--easing-out` |
| Close — scrim | Opacity `1 → 0`, 220ms — slightly slower than modal so the scrim is the last thing to leave |
| Reduced motion | Modal: opacity-only 220ms. Scrim: opacity-only 220ms. No scale. |
| Switch flip | base-ui Switch's built-in transition; visual: thumb slides 22px, 160ms. Reduced motion: instant. |

### 4.12 Modal — accessibility

| Concern | Spec |
|---|---|
| Role | `role="dialog"`, `aria-modal="true"` |
| `aria-labelledby` | On the modal root, pointing to the title `<h3>` id (e.g., `cookie-modal-title`) |
| Focus trap | Active while modal is open (`<dialog>` native trap; reinforced by `focus-trap-react` for older browsers). |
| Initial focus | Lands on **the first togglable switch (Analytics)**, NOT the close button. Per Step 3 of the brief — "land focus where the user makes a decision, not on dismissal." |
| Switch a11y | `role="switch"`, `aria-checked={true|false}`, `aria-label` matching the row label, `aria-describedby` pointing to the description paragraph |
| Necessary row | `aria-disabled="true"`, `aria-checked="true"`. Reads to AT as "Strictly necessary, on, dimmed, required" — the "Always on" badge is `<span aria-hidden="true">` (decorative; AT reads the disabled/checked state). |
| ESC | Closes the modal AND returns focus to the "Manage preferences" button on the banner. The banner re-traps focus. (`<dialog>` native ESC fires `close` event.) |
| Backdrop click | Does NOT close the modal — `onClick` on the scrim is a no-op. The native `<dialog>` `cancel` event is preventDefault'd. |
| Touch targets | Switch hit-area extended to 48×48 (transparent padding); close button 48×48; footer buttons 48 tall. |
| Contrast pairs | Modal title (#1A1A1A) on white: **18.9 : 1** ✓. Description (#4A4A4A) on white: **8.9 : 1** ✓. Switch off-track (#C9C0AE) on white: **3.0 : 1** ✓ non-text. Switch on-track (#4D8A3F) on white: **4.7 : 1** ✓ non-text. |
| Focus rings | Switch focus: 2px solid `--color-focus-ring`, offset 3px (clears the 12px rounded track). Buttons: standard 2px ring offset 2px. Close button: 2px ring offset 2px. |

## 7. Decisions made in-phase (for Sunset-Services-Decisions.md)

Three calls Design had to make beyond the brief. All three are surfaced to Chat in §8 below for ratification (no silent ratifications per Project Instructions §4).

**DM-1 — Personalization grouping.** Brief specifies that `ad_user_data` and `ad_personalization` map to one user-facing toggle. Decision made: row is named "Personalization" (more accessible to non-technical users than "Ad data + personalization"); when the toggle is **on**, BOTH signals fire as `granted`; when **off**, BOTH fire as `denied`. They are never split. Rationale: a non-technical user has no mental model for sharing data without using it. Code wires this in B.03.

**DM-2 — Analytics default state in the modal.** Brief specifies the four categories but not the default state of each. Decision made: Necessary is locked-on (per CMv2 spec); **Analytics defaults to ON; Marketing defaults to OFF; Personalization defaults to OFF.** Rationale: pure-analytics signals (GA4 page-views) are the lowest-privacy-cost category and the highest-value-to-Sunset category. Marketing and personalization have stronger user expectation of opt-in. Phase M.04 GDPR/CCPA review may flip Analytics to default-OFF; flagged as OQ-3.

**DM-3 — Banner primary action color.** Brief says "cookie banner uses green for primary." Locked. Confirmed not to use amber. Decision made for secondary and tertiary: Reject all = bordered green secondary; Manage preferences = ghost. Rationale: Reject all is functionally equivalent in weight to Accept all (a user choosing "don't track me" is making just as valid a primary choice), but bordered green visually de-emphasizes vs primary fill to honor the "Accept all is the recommended path" implicit pattern most users expect, while still keeping Reject all reachable and clearly affirmative.

**DM-4 — Mockup format.** Brief specifies "embed PNG/JPG exports inline OR link a Figma file with view-only access AND attach static exports." Decision made: **inline HTML/CSS mockups in a sibling file** (`Phase-B-02-Legal-Mockups.html`). Rationale: a) Code can inspect every element to read padding/font directly — higher fidelity than a PNG, b) no external tool access needed (matches brief's "without external tool access" requirement), c) renders identically to the eventual implementation since both use Tailwind v4 CSS variables. Flagged so Chat can confirm the format is acceptable; if not, exports can be screenshotted from the HTML file in one pass.

## 8. Open questions for Chat / Code

**OQ-1 — Termly iframe vs inline render.** Termly's embed is currently inline (renders policy HTML directly into our DOM, which is what the §3 CSS override targets). Termly has historically experimented with iframe rendering for stability. If Termly switches to iframe before B.03 ships, the §3 CSS block cannot reach inside — Code must inject the same block via Termly's dashboard `Custom CSS` field (which Termly applies inside the iframe) AND remove our DOM-side `.prose` overrides. **Action:** Cowork verifies inline-vs-iframe at signup. If iframe, the override block in §3 still works because Termly injects it inside; no design change needed.

**OQ-2 — Hiding Termly's branded footer on free tier.** Section 3.1 hides `.termly-footer { display: none }`. Termly's free-tier TOS may require their branded footer remain visible. **Action:** Cowork checks the free-tier TOS during signup. If hiding is disallowed, remove that one rule from the override block. The branded footer is small and our `--color-text-muted` already neutralizes it visually.

**OQ-3 — Analytics default state.** See DM-2. Recommend Chat review: should Analytics default ON, or default OFF? GDPR strictest reading argues default-OFF for ALL non-necessary; CCPA + most US-state laws allow default-ON for analytics. We're an Illinois-serving site. Default-ON keeps GA4 data quality high (most users will not visit the modal); default-OFF is the safer compliance posture if our jurisdiction expands. **Recommendation: stay at default-ON for v1; revisit with legal review in M.04.**

**OQ-4 — Footer "Cookie preferences" link.** The brief implies but does not specify a sitewide footer link to reopen the preferences modal after a user has made an initial choice. Decision: add `cookies.footer.managePrefs` link in the footer's secondary row, which on click reopens the same modal without re-showing the banner. Already in §6 strings table. **Action:** Confirm with Chat; if rejected, drop the key. Recommended kept — most sites require a "manage preferences" entry point post-consent for GDPR/CCPA.

**OQ-5 — Mockup format.** See DM-4. Confirm HTML/CSS mockups + linked file are acceptable in lieu of PNG exports.

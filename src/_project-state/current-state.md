## What works (Phase 2.13 additions)

- **`POST /api/webhooks/servicem8`** is live and reachable. With `SERVICEM8_ENABLED=false` (the default — ServiceM8 not yet adopted), the route returns 200 + `{status:'simulated',reason:'feature-flag'}` for any request body, with any headers, without parsing, signature-checking, or persisting. Verified end-to-end against `localhost:3037` via the synthetic test battery.
- **HMAC-SHA256 signature verification** when the flag is on. The route reads the raw request bytes (NOT JSON.parse output — round-tripping would rewrite keys/whitespace and break the HMAC) and compares `createHmac('sha256', SERVICEM8_WEBHOOK_SECRET).update(rawBody,'utf8').digest('hex')` against the `x-servicem8-signature` header using `timingSafeEqual` with a length-mismatch guard. Mismatched or missing signature → 401 + `{status:'error',reason:'invalid-signature'}` with NO Sanity write. Header name + signature format are assumptions pending confirmation when Erick configures the real ServiceM8 webhook (`x-servicem8-signature`, hex digest of the raw body).
- **Zod-validated payload schema.** The route Zod-validates the parsed JSON against `serviceM8WebhookSchema` — required `eventId`, literal `eventType: 'job.completed'`, required `jobId`, ISO-8601 with offset `occurredAt`, free-form `data` record. Validation failure → 400 + `{status:'error',reason:'invalid-payload'}` with the Zod error tree NOT echoed back (opaque on purpose — ServiceM8 doesn't need our parse errors and surface area minimization is the priority). Invalid JSON → 400 + `{status:'error',reason:'invalid-json'}`. Off-spec: root schema uses default strip mode (NOT the plan's `.passthrough()`) because Zod 3.25's type inference collapses known keys to `unknown` when `.passthrough()` is on; plan rationale is still satisfied because Zod's default doesn't reject extras either (silent-strip). Logged in `Sunset-Services-Decisions.md`.
- **Deterministic-ID dedup.** Each valid event lands at `servicem8Event-<slug(eventId)>` (slug pass strips characters Sanity rejects in `_id` — whitespace, slashes, etc.). First send → 200 + `{status:'ok',sanityDocId}`. Replayed send with the same body → 200 + `{status:'ok',deduped:true,sanityDocId}` with no second write. Sanity failure on first send → opaque 500 + `{status:'error',reason:'persist-failed'}` (error logged server-side, not in the response body, since ServiceM8 retries on 5xx and leaking internals is unsafe).
- **`servicem8Event` Sanity document type** visible in `sunsetservices.sanity.studio` left navigation under "ServiceM8 Event". 3 field groups (event / processing / meta). Event group: eventId, eventType, jobId, payload (raw JSON blob), signatureValid (always true at write time), receivedAt. Processing group: status enum (pending/processing/completed/failed/skipped, default pending), processedAt, processingError, relatedProjectId (reference to `project`), telegramApprovalState enum (not_sent/pending/approved/rejected, default not_sent). Meta group: createdAt, lastUpdatedAt. Preview shows `<eventType> — <jobId>` with `<status> · <receivedAt>` subtitle. Ordered by `receivedAt desc`.
- **Verification battery (`scripts/test-servicem8-webhook.mjs`)** is committed as a reusable harness. The route upserts a `quoteLeadPartial` document at deterministic ID `quoteLeadPartial-<sessionId>` — fetch-then-patch-or-create preserves `firstSeenAt` while advancing `lastUpdatedAt` and `lastStepReached`. When a matching full `quoteLead` later lands, the partial's `converted` field flips to `true`. NEVER fires on Step 4→5 (PII boundary — Step 4 is the first step that touches name/email/phone/address, and the partial endpoint's Zod schema rejects PII fields too as a defensive backstop).
- **`WIZARD_SUBMIT_ENABLED=false` master kill switch.** Both routes return 200 + `status: 'simulated'` with zero side effects — Sanity untouched, Resend not called, Mautic stub log line not fired, no error logging. The wizard treats the response identically to a real success and redirects to `/thank-you/`, so the route remains demoable with the backend intentionally off.
- **Honeypot anti-spam on Step 5.** A visually hidden `#company_website` input (off-screen positioning, `aria-hidden="true"`, `tabIndex={-1}`, `autoComplete="off"`) gets a populated value when a naive bot fills every input. The route handler checks for this BEFORE Zod runs and returns silent 200 — bot doesn't learn which field tripped it. (The previous approach, putting `honeypot: max(0)` inside the Zod schema, was rejected during smoke testing because the 400 error body exposed the field name.)
- **Two new Sanity document types in the Studio.** "Quote Lead" (8 contact + 6 project + 6 meta fields, ordered by submittedAt desc) and "Quote Lead — Abandoned" (sessionId + timestamps + last-step + converted flag, ordered by lastUpdatedAt desc). Both deployed via `npm run studio:deploy`.
- **Mautic stub gated on `MAUTIC_ENABLED`.** Both `pushFullLeadToMautic()` and `pushPartialLeadToMautic()` log a single no-op line and return `{synced: false}` at Phase 2.06. Flipping `MAUTIC_ENABLED=true` + populating `MAUTIC_BASE_URL` + `MAUTIC_API_KEY` switches them to the real implementation (TODO block marked `Phase 2.x`) without any other code edits at the call site.
- **Session ID linkage between partials and full submits** via a client-side localStorage UUID at key `sunset_wizard_session_id`. Generated via `crypto.randomUUID()` with a tiny RFC4122 v4 fallback for environments without the API. Cleared on successful submit.
- **Six new env vars** added to `.env.local`, `.env.local.example`, and Vercel (Production + Preview): `WIZARD_SUBMIT_ENABLED=true`, `MAUTIC_ENABLED=false`, `MAUTIC_BASE_URL=`, `MAUTIC_API_KEY=`, `RESEND_FROM_EMAIL=onboarding@resend.dev`, `RESEND_TO_EMAIL=info@sunsetservices.us` (overridden to `dinovlazar2011@gmail.com` at Phase 2.06 dev window per the 2026-05-12 Resend TO routing decision). `WIZARD_SUBMIT_ENABLED` was previously `false` on Vercel (Phase 2.02 default) — flipped to `true` as part of this phase.

- **`zod@^3.25.x` pinned** (resolved from the spec's `^3.23` range; functionally equivalent). Server-only — the Zod imports never reach the client bundle.
